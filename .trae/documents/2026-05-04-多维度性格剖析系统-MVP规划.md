# 多维度性格剖析系统（占星 + 塔罗 + MBTI）MVP 规划

## Summary

目标：做一个 Web Demo/MVP，在不接入 LLM 的前提下，完成「出生信息 → 星盘计算 → 抽取塔罗牌义 → 融合成结构化报告」的端到端流程；并预留后续接入 LLM 的接口与数据形态。

本规划基于当前仓库现状（仅有 README.md）与以下已确认偏好：

- 交付形态：Web 应用
- LLM：第一版不接入（先跑通数据与规则链）
- 技术栈：由我推荐并在本规划中锁定
- 占星粒度：完整星盘（kerykeion 计算）
- 塔罗：牌阵可配置（至少支持单张/三张）
- 数据来源：Kaggle 数据集 + GitHub 仓库（你提供的 clone / download 线索）
- 出生地点：第一版使用内置城市库（离线可用、覆盖可逐步扩展）

## Current State Analysis

- 仓库内容：仅 [README.md](file:///workspace/README.md)
- 未存在任何运行时、依赖管理、服务端/前端结构、数据目录与脚本

这意味着 MVP 需要从 0 初始化工程结构与最小可运行链路。

## Proposed Changes

### 1) 技术栈锁定（推荐方案）

推荐使用「前后端分离」：

- 后端：Python + FastAPI
  - 理由：数据集解析、占星计算（kerykeion）、后续 LLM 编排更自然
- 前端：React + Vite + TypeScript
  - 理由：开发体验成熟、组件生态好，适合快速做表单 + 报告展示

备选方案（不选用）：Next.js 全栈一体（好部署但占星/数据处理在 JS 侧会更绕或需要 Python 子服务）。

### 2) 目录与模块边界（面向后续 Agent 化）

建议初始化为单仓库，逻辑边界清晰，便于后续把“分析能力”抽成工具/Agent：

- `apps/web/`：前端（输入向导、报告页、历史记录本地保存）
- `services/api/`：后端（星盘计算、抽牌、牌义检索、报告生成）
- `data/`
  - `data/raw/`：下载的原始数据（Kaggle zip、csv/json 等）
  - `data/processed/`：清洗后的结构化数据（统一为 json）
- `packages/domain/`：可选（若需要共享 schema / 模板，可放到独立包）
- `scripts/`：一次性/运维脚本（下载 Kaggle、导入城市库、数据校验）

### 3) 数据集与依赖引入策略

你已提供的线索：

- Kaggle：`morrispoint/complete-tarot-card-meanings-all-78-cards`
- GitHub：
  - `PKU-YuanGroup/Machine-Mindset`（用于 MBTI/人格相关文本或规则参考）
  - `g-battaglia/kerykeion`（占星计算）
  - `jimmc414/onefilellm`（后续可用于打包知识）
  - `plastic-labs/steerability-eval`（可选：后续评测用，MVP 阶段不强依赖）

MVP 原则：

- 依赖优先走包管理（pip/npm）而不是直接把第三方仓库源码 vendoring 进来
- 数据集统一落地到 `data/raw`，清洗后落地到 `data/processed`，后端只读 `processed`
- 为每个数据源建立“下载器/解析器”适配层，后续换数据集不改业务逻辑

### 4) Kaggle 下载与解析（可复用的 Data Pipeline）

后端增加一个数据流水线模块（不在请求链路中运行，避免 Web 请求时做重活）：

1. 下载：
   - 使用 Kaggle CLI 下载 tarot 数据集 zip 到 `data/raw/kaggle/`
   - 需要你在本地/部署环境提供 Kaggle token（`~/.kaggle/kaggle.json` 或环境变量）
2. 解压与规范化：
   - 将 78 张牌解析为统一 schema，例如：
     - `id`、`name`、`arcana`（major/minor）、`suit`、`number`
     - `upright_keywords`、`reversed_keywords`
     - `meaning_upright`、`meaning_reversed`
     - `love`、`career`、`finance` 等细分（若数据集包含）
   - 输出为：`data/processed/tarot/cards.json`
3. 校验：
   - 必须校验数量=78、字段完整性、名称唯一性

### 5) 城市库（离线地点获取）

目标：用户输入出生城市名，系统在离线城市库中查到经纬度与时区信息，用于星盘计算。

设计：

- 数据形态：`data/processed/geo/cities.json`
  - 字段示例：`name`、`name_en`、`country`、`admin`、`lat`、`lon`、`timezone`
- MVP 覆盖：
  - 先内置“常用城市集”（比如中国主要城市 + 你目标用户常见地区）
  - UI 里提示：若未匹配到城市，提供“手动输入经纬度+时区”的兜底入口（但默认不强制）

数据来源策略（MVP 不强约束具体来源）：

- 你可提供一份城市表（csv/json），我负责清洗入 `processed`
- 或后续用公开数据源导入（如 geonames 派生数据），但需要明确许可与字段映射

### 6) 核心业务流程（MVP 请求链路）

前端收集：

- 必填：出生日期、出生时间、出生城市、性别（可选）
- 可选：关注领域（爱情/事业/自我成长等）、MBTI（自填或选择）
- 可选：塔罗牌阵选择（单张/三张）

后端核心接口建议：

- `POST /api/analyze`
  - 输入：用户出生信息 + 领域 + 牌阵配置
  - 输出：结构化报告 JSON（同时包含用于 UI 展示的字段）

后端在一次请求中完成：

1. 地点解析：城市名 →（lat, lon, timezone）
2. 星盘计算：调用 kerykeion 生成结构化星盘数据（行星、宫位、相位、星座落点等）
3. 塔罗抽牌：
   - 从 78 张牌中按牌阵抽取 N 张
   - 支持正位/逆位（可按配置概率或用户选择）
4. 报告生成（无 LLM，规则与模板驱动）：
   - 把“星盘特征 → 性格段落/标签”映射为可读文本
   - 把“抽到的牌 → 牌义（按领域/正逆）”拼接为当前状态与建议
   - 把 MBTI（若提供）作为一个“修饰层”，补充偏好与沟通风格段落

### 7) 报告结构（前后端统一 Schema）

建议固定输出结构，后续接 LLM 时保持字段稳定（LLM 只负责丰富文案与场景化）：

- `profile`
  - `summary`：一句话画像
  - `traits`：若干标签（例如 6-10 个）
  - `strengths` / `blind_spots`
- `astrology`
  - `raw_chart`：星盘原始结构（用于调试/二次分析）
  - `highlights`：关键信息提要（比如“太阳/上升/月亮”“主宰星”“强相位”）
  - `interpretation`：解释文本（模板生成）
- `tarot`
  - `spread`：牌阵定义
  - `cards[]`：每张牌（名称、正逆、领域解释、关键词）
  - `interpretation`：综合解读
- `mbti`
  - `type`：用户输入
  - `interpretation`
- `action_guide[]`：3-5 条可执行建议
- `followup_questions[]`：引导继续追问的问题

### 8) “可扩展到 Agent” 的预留点

MVP 不接 LLM，但要为后续 Agent/对话解读预留接口：

- 把“输入 → 中间结构 → 输出”拆成可复用函数/服务：
  - `compute_chart(input) -> chart`
  - `draw_tarot(spread, seed?) -> cards`
  - `generate_report(chart, cards, mbti, domain) -> report`
- 统一数据结构与版本号：
  - 对 `report` schema 增加 `schema_version`
  - 对数据集增加 `dataset_version` 与校验摘要（hash）

## Assumptions & Decisions

- Python 与 Node 运行环境可用（用于 FastAPI 与 Vite）
- Kaggle 下载在开发机/部署机上可配置 token；MVP 阶段不把 token 写入仓库
- kerykeion 以“包依赖”方式引入；若遇到版本/兼容问题，再回退到 clone + 本地引用
- 城市库采用离线 json 并支持逐步扩充；未命中时提供手动输入兜底
- MBTI 第一版以“用户自填”为主，不做自动推断（避免早期规则争议）

## Verification (Acceptance Criteria)

MVP 验收以“可运行 + 输出稳定 + 数据可复现”为主：

1. 能完成一次端到端流程：
   - 前端填写出生信息 + 选择牌阵 + 提交
   - 后端返回结构化报告，前端正确渲染
2. 数据集流水线可重复执行：
   - Kaggle tarot 数据能下载、解析、输出 `cards.json`
   - 校验通过（78 张牌，字段齐全）
3. 星盘计算可验证：
   - 对至少 3 个已知样例输入（你提供或我们共同确认）输出稳定
4. 牌阵可配置：
   - 单张与三张都能抽取并渲染
